Production pipeline

Chapter alignment from EPUB + audio to timestamped canonical words

This is the real Podible pipeline behind chapter_analysis: how an EPUB and an audio asset become chapter timings plus transcript words that are projected back onto the canonical EPUB chapter structure.

What the job produces

Chapter start/end times

Stored as chapter timing JSON for /chapters/:assetId.json.

Canonical transcript words

Transcript timestamps projected back onto EPUB wording instead of raw Whisper wording.

Per-chapter transcript segments

Stored in transcript JSON so the app can show chapter-level aligned text.

Key files

  • src/library/chapter-analysis.ts
  • src/library/transcript-analysis-pool.ts
  • src/library/transcript-analysis-worker.ts
  • src/library/media.ts

The heavy alignment core now runs in worker threads, while EPUB loading, transcription, DB writes, and job orchestration stay in the main worker process.

High-level flow

1

Queue selection

Podible only queues the job when a book has both a preferred audio asset and a preferred EPUB asset. The worker later refuses to run if the target audio is no longer the preferred audio asset.

2

Fingerprint gate

Before doing any expensive work, the job computes a chapter fingerprint from the audio + EPUB inputs and a transcript fingerprint from the audio files. A prior success is reused only when the stored fingerprints and algorithm_version still match.

3

EPUB parsing

Podible loads the EPUB spine and TOC, strips HTML, tokenizes each chapter, computes cumulative word counts, and normalizes chapter ordering into a list of canonical chapter entries.

4

Whole-book transcript acquisition

If there is no cached transcript for the current transcript fingerprint, Podible splits the audio into overlapping chunks, extracts clips, calls Whisper, and merges the chunk transcripts into a single ordered word stream with timestamps.

5

Pure transcript analysis

The CPU-heavy alignment phase runs in a worker thread through runTranscriptAnalysis(...). This phase consumes only serializable inputs: EPUB entries, transcript words, duration, glossary terms, and chunk plans.

6

Persist canonical outputs

The main thread stores the aligned transcript payload and chapter timing payload back into SQLite. From there, /transcripts/:assetId.json and /chapters/:assetId.json can serve the aligned output without recomputing.

Inside the pure analysis phase

Inputs

Canonical EPUB chapters, transcript words with timestamps, duration, glossary terms, chunk plan metadata.

Boundary probes

Generate candidate snippets around EPUB chapter boundaries and estimate when those boundaries should occur.

Best probe alignment

Search transcript windows near each estimate and score fuzzy probe matches to resolve real boundary times.

↓

Interpolation

Use resolved boundaries to assign a start and end time to every EPUB chapter, with ratio-based fallback for gaps.

Chapter segment matching

Within each chapter window, align EPUB tokens to transcript tokens to recover chapter-local canonical words.

Output

Chapter timings, chapter-level transcript segments, boundary counts, and debug/profile metadata.

How canonical EPUB words get timestamps

The raw Whisper transcript is useful because it has timestamps, but its wording is noisy: substitutions, dropped words, extra narration tokens, punctuation drift, and occasional hallucinated filler. Podible therefore treats the EPUB as the canonical text and the transcript as a timestamp source. 

audio chunk -> Whisper transcript words with timestamps
EPUB chapter text -> canonical tokens with no timestamps

alignment:
  match canonical EPUB tokens to nearby transcript tokens
  project transcript start/end times back onto the EPUB token positions

result:
  canonical EPUB words with audiobook timing

This is why the stored transcript payload contains both the merged whole-book transcript and the chapter-aligned canonical segments. The app can show transcript text that actually matches the book instead of exposing raw Whisper output directly.

Alignment algorithm, end to end

The algorithm has two layers. First it resolves chapter boundaries across the whole book. Then it resolves word-level alignment inside each chapter window. The first layer determines where chapters start and end in time. The second layer assigns timestamps to canonical EPUB words.

A

Build chapter boundary probes from EPUB text

For every boundary between chapter n and chapter n+1, Podible builds probe snippets from the tail of the previous chapter and the head of the next chapter. It also computes one or more estimated timestamps for that boundary from EPUB word ratios and, when available, transcript chunk plan hints.

B

Search a transcript window near the estimate

Podible does not scan the entire transcript for every probe. It finds the transcript word index nearest the estimated timestamp, opens a bounded window around that point, and aligns the probe only inside that window. This is why the algorithm stays tractable on long books.

C

Use fuzzy dynamic programming for probe alignment

Probe alignment is a dynamic-programming matrix over EPUB probe tokens and transcript tokens. A diagonal step means “match or substitute these two tokens,” up means “skip the EPUB token,” and left means “skip the transcript token.”

The scoring is intentionally forgiving:

  • exact token match: strong positive score
  • prefix/suffix or small edit-distance match: smaller positive score
  • bad mismatch: negative score
  • skip/delete/insert: mild negative gap penalty

After filling the matrix, Podible backtracks only the best end column, keeps positively scored token pairs, and rejects alignments that fail coverage or drift constraints.

D

Resolve the actual boundary timestamp

Podible tries multiple probe candidates and multiple estimated timestamps. It keeps the best alignment by preferring higher coverage, then more matched tokens, then the result closest to the estimate. This produces a resolved timestamp for the boundary, plus debug data about why it won.

E

Interpolate all chapter timings

Once some boundaries are resolved, Podible interpolates start times for all chapters between those anchors using EPUB cumulative word ratios. That produces a full chapter timing list even when not every raw boundary probe lands.

This is the point where the pipeline shifts from “find confident whole-book anchors” to “build a usable chapter timeline.”

F

Find chapter-local anchors inside each interpolated window

For each chapter, Podible takes the interpolated start/end window and looks for stronger local anchors: the chapter opening, the chapter ending, and a few informative mid-chapter anchor snippets chosen from the EPUB text. Each of those anchors is aligned against the transcript window the same fuzzy way.

G

Fill the gaps between anchors with token-span alignment

Between any two accepted anchors, Podible aligns the EPUB token span and transcript token span with another dynamic-programming pass. This is more local than the whole-book boundary stage: it works inside one chapter and inside one bounded transcript region.

H

Project timestamps back onto canonical EPUB words

Each aligned EPUB word inherits the matched transcript word’s startMs and endMs. EPUB words that do not get a direct transcript partner are filled by interpolation between the nearest timed words, constrained by the chapter start/end window.

The result is a chapter-local stream of canonical EPUB words with timestamps, not just a noisy transcript excerpt.

Why there are two dynamic-programming passes

Pass 1: whole-book boundary resolution

This pass is coarse and estimate-driven. Its job is to answer: “Where in the audiobook does the boundary between these two EPUB chapters probably happen?”

Pass 2: chapter-local word projection

This pass is narrow and anchor-driven. Its job is to answer: “Inside this already-bounded chapter region, which transcript words correspond to which canonical EPUB words?”

Why split them

Doing only a whole-book pass would be too weak for word-level projection. Doing only chapter-local passes would be too fragile because the chapter windows would be unknown. The pipeline needs both.

Stored artifacts

asset_transcripts.transcript_json

Stores the aligned transcript payload: canonical text, canonical words with timestamps, and per-chapter transcript segments.

chapter_analysis.chapters_json

Stores chapter start and end timestamps for the preferred audio asset, which powers the chapter endpoint.

chapter_analysis.debug_json

Stores boundary counts, profiling, chunk counts, glossary data, and other debug information used to inspect alignment quality.

Why worker threads were added

Profiling showed that the expensive part was not SQLite or transcription orchestration. The hot path was pure JavaScript alignment, especially boundary probing and chapter segment matching. Moving just that phase into a small thread pool keeps the rest of the pipeline simple while preventing a handful of long alignment jobs from monopolizing the main worker loop.

Important: the thread boundary is intentionally narrow. The worker thread does not touch the DB, the network, EPUB parsing, or filesystem writes. It only runs the pure transcript analysis function.
