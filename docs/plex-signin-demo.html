Plex Sign-In, The Modern Way

This is the flow Podible should use for browser sign-in: keep one persisted instance keypair, upload the public JWK when requesting a PIN, open Plex Auth in the browser, then let Podible sign a short-lived device JWT to exchange the approved PIN for a Plex JWT and finally create a normal Podible session cookie.

What Matters

Instance keypair Generate it once per Podible instance, not per request and not per user.
Browser starts auth The user still authenticates in a Plex-controlled browser window.
Private key stays on Podible The browser never needs the private key and never has to sign the device JWT.
Podible session stays local After Plex identity is verified, Podible still issues its own session cookie.

Why This Is Better Than The Old Token Flow

The browser still goes through Plex Auth, but Podible no longer needs to treat the old PIN token as the primary trust model.
Podible proves it is the same app instance by signing a short-lived device JWT with its persisted private key.
More moving parts means more implementation work: key generation, JWK storage, deviceJWT signing, and a small browser-side login helper.
